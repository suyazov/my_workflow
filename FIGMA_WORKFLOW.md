# Регламент работы с Figma для вёрстки сайтов и лендингов

Цель: сверстать сайт максимально близко к макету Figma, используя данные Figma Dev Mode/API, экспортированные ассеты и инструментальную проверку. Вёрстка «на глаз» запрещена.

Главное правило: лучше остановиться и сообщить о проблеме, чем заменить элемент макета похожим.

---

## 0. Жёсткие правила для ИИ-агента

Регламент выполняется в режиме pixel-perfect. Агент не имеет права интерпретировать макет творчески.

### 0.1 Источник истины

Источник истины для вёрстки и проверки:

1. Figma node id.
2. Данные Figma Dev Mode/API.
3. Экспортированные из Figma SVG/PNG/WebP.
4. Reference-скриншот Figma.
5. Скриншот сравнения `Figma vs Site`.
6. DOM-замеры.
7. Visual diff.

Эти источники считаются достаточными только при условии, что перед началом вёрстки создан контрольный пакет Figma через `scripts/figma-pull.mjs`.

### 0.2 Главное правило против ложного PASS

DOM-замеры не могут отменять visual diff.

Если возникает противоречие:

```text
DOM-замеры: pass
Visual diff: fail
```

итоговый статус всегда:

```text
НЕ СДАНО
```

Причина: DOM-селекторы покрывают не все элементы, сравнение сделано на неправильном canvas или агент проверил отдельные элементы вместо всей композиции.

Агент обязан:

1. расширить список DOM-селекторов;
2. добавить structural nodes;
3. переснять Figma reference и Site screenshot в одном размере;
4. пересчитать diff;
5. обновить `comparison-report.md`.

Запрещено сдавать работу, если visual diff выше порога, даже если отдельные DOM-замеры находятся в пределах ±4 px.

### 0.3 Запрещено

Запрещено:

- заменять иконки, декоративные элементы, SVG, картинки и формы на похожие;
- использовать элементы из соседних блоков макета;
- менять тип элемента: декоративная иконка не может стать кнопкой, кругом, плюсом, стрелкой или другой фигурой;
- растягивать карточку или секцию на всю ширину, если в Figma у неё задана конкретная ширина;
- менять масштаб всего экрана, чтобы «примерно попасть» в макет;
- убирать `border-radius`, тени, обводки, opacity, gradient, если они есть в Figma;
- менять количество дочерних элементов внутри блока;
- придумывать недостающие элементы;
- заменять сложную векторную фигуру CSS-иконкой, emoji, Font Awesome, Lucide, Heroicons или похожим символом;
- писать «готово», если хотя бы один обязательный элемент блока не сверен с Figma;
- сдавать блок без `figma-reference.png`, `site-result.png`, `diff.png` и `comparison-report.md`;
- объяснять высокий visual diff словами «это шапка», «это WordPress», «это шрифты», если эти зоны не внесены заранее в `visual-diff-ignore.json` и не согласованы.

Если элемент из Figma не получается точно воспроизвести, агент обязан:

1. остановиться;
2. указать `node_id` проблемного элемента;
3. описать, что именно не получилось экспортировать или сверстать;
4. не заменять элемент похожим;
5. поставить статус `НЕ СДАНО`.

Пример критической ошибки:

```text
Figma: зелёный декоративный крест/звезда
Site: белый круг с плюсом
Результат: FAIL. Блок не сдаётся.
```

## 1. Паспорт проекта

Перед применением регламента агент обязан заполнить паспорт проекта. Это делает регламент универсальным для разных проектов: лендингов, многостраничных сайтов, WordPress, Bitrix, Tilda, React, Next.js, статической HTML-вёрстки и других стеков.

```text
Тип проекта:
- лендинг / многостраничный сайт / интернет-магазин / личный кабинет / админка / WordPress / Bitrix / Tilda / React / Next.js / статическая HTML-вёрстка

Источник макета:
- Figma URL:
- FILE_KEY:
- основной NODE_ID:
- есть ли desktop-макет: да/нет
- есть ли tablet-макет: да/нет
- есть ли mobile-макет: да/нет

Целевые разрешения:
- desktop:
- tablet:
- mobile:

Стек проекта:
- CMS/Framework:
- где лежат стили:
- где лежат шаблоны:
- где лежат изображения:
- способ деплоя:

Что важнее при конфликте:
1. ТЗ заказчика.
2. Актуальный текст/контент.
3. Figma-макет.
4. Текущая реализация сайта.

Допустимые отклонения:
- desktop: ±4 px
- mobile: ±6 px, если нет отдельного mobile-макета
- текст: допускается перенос строк, если не ломает композицию
- изображения: допускается WebP вместо PNG/JPG при визуальном совпадении

Что нельзя менять без согласования:
- структуру блоков;
- порядок секций;
- тексты;
- контакты;
- формы;
- CTA-кнопки;
- декоративные элементы;
- иконки;
- изображения;
- смысловые акценты.
```

Если паспорт проекта не заполнен, агент не имеет права начинать работу.

## 2. Подготовка проекта

### 2.1 Необходимые данные

Перед работой нужны:

- **Figma file key** — из URL макета: `https://www.figma.com/design/<FILE_KEY>/<NAME>?node-id=<NODE_ID>`;
- **Figma API token** — личный токен разработчика;
- **Node ID целевого артборда** — из `node-id=...` в URL или Dev Mode;
- **Node ID каждого крупного блока** — header, hero, about, services, benefits, cases, footer и т.д.;
- **URL тестового сайта** для проверки результата;
- **локальный путь проекта** или SSH-доступ к серверу.

### 2.2 Инструменты

Обязательные инструменты:

- `node` 18+;
- `scripts/figma-pull.mjs` — выгрузка контрольного пакета Figma;
- `curl` + `jq` — дополнительные запросы к Figma API;
- `puppeteer` или `chromium-browser` — скриншоты и DOM-замеры;
- `pixelmatch`, `looks-same` или аналог — visual diff;
- `sharp`, `cwebp` или ImageMagick — оптимизация изображений;
- DevTools браузера — проверка computed styles.

### 2.3 Рабочая папка проверки

Перед началом вёрстки должна быть создана папка:

```text
figma-audit/
  project-passport.md
  figma_node_raw.json
  figma_manifest.json
  figma_image_fills_raw.json
  figma_assets_report.json
  assets/*.svg
  assets/image-fill-*.png
  reference/figma-reference-<NODE_ID>.png
  figma_elements.json
  figma_blocks_manifest.json
  figma_assets_manifest.json
  layout-skeleton.json
  visual-diff-ignore.json
  site-result.png
  diff.png
  dom-measurements.json
  hero-layout-report.md
  comparison-report.md
```

Если этих файлов нет, работа не считается проверенной. Исключение: `visual-diff-ignore.json` может отсутствовать только если нет игнорируемых зон.

## 3. Обязательная выгрузка контрольного пакета из Figma

### 3.1 Правило

Перед началом вёрстки агент обязан выгрузить контрольный пакет Figma через `scripts/figma-pull.mjs`.

Без контрольного пакета агент не имеет права:

- начинать вёрстку;
- заменять иконки похожими элементами;
- пересобирать сложные SVG через CSS;
- сдавать результат как готовый.

### 3.2 Запуск

```bash
FIGMA_TOKEN="figd_..." \
FILE_KEY="cU6O3Xlsar8HPCleslQ11O" \
NODE_ID="22:2" \
OUT_DIR="./figma-audit" \
node scripts/figma-pull.mjs
```

`NODE_ID` можно передавать как `22:2` или как из URL Figma: `22-2`.

### 3.3 Что должен выгрузить скрипт

```text
figma-audit/
  figma_node_raw.json
  figma_manifest.json
  figma_image_fills_raw.json
  figma_assets_report.json
  assets/*.svg
  assets/image-fill-*.png
  reference/figma-reference-<NODE_ID>.png
```

Назначение файлов:

| Файл | Назначение |
|---|---|
| `figma_node_raw.json` | Сырая структура целевого node id из Figma API |
| `figma_manifest.json` | Полный список элементов: id, name, type, размеры, координаты, цвета, шрифты, children |
| `figma_image_fills_raw.json` | Список image fills из Figma |
| `figma_assets_report.json` | Отчёт о выгруженных SVG/PNG и ошибках экспорта |
| `assets/*.svg` | Экспортированные VECTOR / GROUP / INSTANCE / COMPONENT |
| `reference/figma-reference-<NODE_ID>.png` | Эталонный рендер фрейма из Figma |

### 3.4 Fail-gate контрольного пакета

Если в `figma_assets_report.json` есть ошибки, агент обязан остановиться.

Запрещено исправлять ошибку экспорта так:

- взять похожую иконку;
- нарисовать похожий плюс/крест CSS-ом;
- заменить декоративный SVG на Font Awesome, Lucide, Heroicons или emoji;
- использовать элемент из соседнего блока.

Критическое правило:

```text
Нет SVG/PNG/WebP из Figma → нет права верстать этот элемент вручную.
Нет node id → нет права придумывать элемент.
Нет reference PNG → нет права сдавать результат.
```

## 4. Реестр элементов, manifest блоков и layout skeleton

### 4.1 Полный реестр элементов

Перед вёрсткой должен быть создан `figma_elements.json`. Он может быть копией или очищенной версией `figma_manifest.json`.

В реестр обязательно включать:

- `id` — Figma node id;
- `name` — имя слоя;
- `type` — `TEXT`, `FRAME`, `GROUP`, `RECTANGLE`, `VECTOR`, `INSTANCE`, `COMPONENT`, `ELLIPSE`, `LINE`, `BOOLEAN_OPERATION`, `STAR`, `POLYGON`;
- `characters` — текст, если это `TEXT`;
- `absoluteBoundingBox`;
- `absoluteRenderBounds`;
- `relative` — координаты относительно целевого артборда;
- `fills`;
- `strokes`;
- `strokeWeight`;
- `cornerRadius`;
- `rectangleCornerRadii`;
- `opacity`;
- `effects`;
- `layoutMode`;
- `itemSpacing`;
- `paddingLeft`, `paddingRight`, `paddingTop`, `paddingBottom`;
- `style` — `fontFamily`, `fontSize`, `fontWeight`, `lineHeightPx`, `letterSpacing`;
- `componentId`;
- `children_count`;
- `children_ids`.

Нельзя фильтровать только `TEXT`, `RECTANGLE`, `FRAME`. Нужно собирать также `GROUP`, `VECTOR`, `INSTANCE`, `COMPONENT`, `BOOLEAN_OPERATION`, `STAR`, `POLYGON`, иначе агент потеряет декоративные элементы и может заменить их выдуманными.

### 4.2 Manifest блоков

Каждый крупный блок должен иметь manifest в `figma_blocks_manifest.json`.

Пример:

```json
{
  "blocks": [
    {
      "block_name": "Кому подходит",
      "figma_node_id": "xxx:yyy",
      "role": "section/card",
      "required_children": [
        {
          "node_id": "xxx:1",
          "name": "Кому подходит",
          "type": "TEXT",
          "required": true
        },
        {
          "node_id": "xxx:4",
          "name": "decorative-icon",
          "type": "VECTOR",
          "required": true,
          "export_as": "svg"
        }
      ],
      "forbidden_replacements": [
        "white circle plus",
        "font icon",
        "emoji",
        "icon from another block"
      ]
    }
  ]
}
```

Если в DOM после вёрстки количество элементов, типы или визуальные роли отличаются от manifest — работа считается неготовой.

### 4.3 Layout skeleton

Перед правкой агент обязан зафиксировать layout skeleton каждого ключевого экрана.

Для hero обязательно проверяются не только тексты и кнопки, но и structural nodes:

- body/page background;
- header;
- hero section;
- общий container;
- левая колонка;
- правая колонка;
- grid/flex layout;
- gap между колонками;
- image frame;
- CTA button frame;
- карточки/бейджи;
- декоративные элементы.

Для каждого structural node фиксируются:

```text
node_id
name
type
x
y
width
height
display role
background
border-radius
padding
gap
children_count
```

Если structural node не попал в проверку, DOM-замеры считаются неполными.

Запрещено писать «DOM-замеры совпадают ±4 px», если проверялись только текстовые элементы, а не контейнеры.

## 5. Правило работы с SVG, иконками, декоративными элементами и изображениями

### 5.1 SVG и декоративные элементы

Все декоративные элементы из Figma должны быть либо:

1. экспортированы из Figma как SVG/PNG/WebP;
2. либо воссозданы CSS только если это простая фигура и все её параметры явно извлечены из Figma.

Правило по типам:

| Тип в Figma | Как верстать |
|---|---|
| `VECTOR` | Экспортировать как SVG |
| `INSTANCE` | Экспортировать как SVG или PNG |
| `COMPONENT` | Экспортировать как SVG или PNG |
| `GROUP` со сложной геометрией | Экспортировать как SVG |
| `BOOLEAN_OPERATION` | Экспортировать как SVG |
| `STAR` / `POLYGON` | Экспортировать как SVG |
| Простая `RECTANGLE` | Можно CSS, если известны точные размеры, цвет, radius, opacity, stroke |
| Простая `ELLIPSE` | Можно CSS, если известны точные размеры, цвет, opacity, stroke |

Запрещено заменять:

- зелёный декоративный крест на белый плюс;
- декоративную иконку на кнопку;
- SVG из макета на Font Awesome / Lucide / Heroicons;
- SVG из макета на emoji;
- SVG из макета на похожий CSS-рисунок;
- один декоративный элемент на другой декоративный элемент из соседнего блока.

Если в макете стоит зелёная декоративная иконка, на сайте должна быть именно она: тот же цвет, размер, позиция, opacity и форма.

Если элемент не найден или не экспортируется, нужно оставить технический комментарий в отчёте:

```text
НЕ СДАНО: не удалось экспортировать node_id=xxx:yyy, элемент не заменялся.
```

### 5.2 Изображения

Для каждого изображения из Figma агент обязан сверить:

- node id image frame;
- размер frame;
- `object-fit`;
- crop/focal point;
- border-radius;
- opacity;
- effects;
- положение внутри контейнера;
- фактический используемый файл;
- совпадение с image fill из Figma.

Если изображение визуально похоже, но имеет другой crop, размер, масштаб или позицию — это `FAIL`.

## 6. Внесение правок

### 6.1 Принципы

Источник истины — только Figma node id, данные Dev Mode/API и выгруженный контрольный пакет.

Перед изменением блока агент обязан:

1. определить node id блока;
2. выгрузить все дочерние элементы блока;
3. составить список обязательных элементов;
4. сверить, что в реализации присутствуют все элементы из Figma;
5. убедиться, что не добавлены лишние элементы;
6. проверить, что decorative elements экспортированы или точно восстановлены;
7. проверить, что блок не изменил свою визуальную роль;
8. проверить, что layout skeleton совпадает с Figma.

Для каждого блока фиксируются:

- ширина;
- высота;
- x/y позиция внутри артборда;
- `border-radius`;
- background;
- padding;
- gap;
- расположение всех текстов;
- расположение всех иконок;
- цвета;
- шрифты;
- размеры;
- line-height;
- opacity;
- stroke;
- effects.

Карточка из Figma не должна превращаться в полноширинную секцию.

Если в Figma блок имеет скругление `12px`, ширину `750px` и высоту `163px`, агент не имеет права сделать прямоугольник `1200px` без скругления.

### 6.2 Сохранение состава блока

Для каждого блока должно выполняться соответствие:

```text
Figma block children === Site block children
```

Допустимые отличия:

- технические wrapper-элементы, если они не меняют визуальный результат;
- адаптивные переносы текста, если нет отдельного mobile-макета;
- оптимизированный формат изображения, если визуально изображение совпадает.

Недопустимые отличия:

- отсутствие элемента из Figma;
- добавление визуально заметного элемента, которого нет в Figma;
- замена `VECTOR` на другой символ;
- замена декоративной формы на кнопку;
- замена формы, цвета или opacity декоративного элемента;
- изменение роли блока: карточка стала секцией, бейдж стал кнопкой, иконка стала интерактивным элементом.

### 6.3 Cache-busting

При любой смене файла:

- CSS/JS: менять query-string: `style.css?v=3` → `style.css?v=4`;
- изображения: класть под новым именем: `hero-figma-v2.webp` вместо перезаписи `hero.webp`;
- SVG: при изменении класть под новым именем: `icon-cross-v2.svg`.

После деплоя нужно проверить, что браузер получает новую версию файла.

## 7. Проверка результата

### 7.1 Правило единого масштаба сравнения

Перед сравнением агент обязан привести Figma reference и скриншот сайта к одному canvas.

Обязательные условия:

- ширина reference PNG из Figma должна совпадать с шириной browser viewport;
- `deviceScaleFactor` должен быть `1`;
- zoom браузера должен быть `100%`;
- сайт должен сниматься на том же разрешении, что и Figma frame;
- запрещено сравнивать скриншоты разного размера;
- запрещено делать выводы по скриншотам, которые были масштабированы в чате, браузере или markdown-превью.

Если размеры файлов разные, сравнение считается невалидным.

Пример:

```text
Figma reference: 1440×900
Site screenshot: 1379×664
Результат: FAIL. Сравнение невалидно, нужно переснять на одинаковом canvas.
```

### 7.2 Скриншоты

Минимальный набор скриншотов перед сдачей:

```text
figma-reference-desktop.png
site-result-desktop.png
diff-desktop.png
figma-reference-mobile.png
site-result-mobile.png
diff-mobile.png
```

Если mobile-макета в Figma нет, это нужно указать в отчёте.

### 7.3 DOM-замеры

DOM-замеры должны сохраняться в `dom-measurements.json`.

Проверять нужно:

- coordinates `x/y`;
- ширину и высоту;
- structural nodes;
- шрифты;
- размеры текста;
- line-height;
- цвет текста;
- фон;
- border-radius;
- opacity;
- наличие/отсутствие лишних элементов.

Цель: разница не более 2–4 px по ключевым элементам первого экрана и не более 4 px по ключевым элементам остальных блоков.

Исключение: адаптивные состояния могут иметь другую геометрию, если для них нет отдельного макета. Но подмена элементов запрещена всегда.

### 7.4 Visual diff как fail-gate

Visual diff является обязательным критерием сдачи.

Пороговые значения:

```text
0–3%        pass
3–7%        warning, нужна ручная проверка и объяснение
больше 7%   FAIL
больше 10%  критический FAIL
```

Если visual diff больше 7%, агент не имеет права писать «готово», даже если DOM-замеры отдельных элементов совпадают.

Если агент считает часть отличий допустимой, он обязан заранее создать файл:

```text
visual-diff-ignore.json
```

В нём должны быть перечислены зоны, которые можно игнорировать:

```json
{
  "ignored_regions": [
    {
      "name": "dynamic admin bar",
      "x": 0,
      "y": 0,
      "width": 1440,
      "height": 32,
      "reason": "WordPress admin bar, отсутствует в Figma",
      "approved": true
    }
  ]
}
```

Запрещено объяснять высокий diff после проверки словами «это шапка», «это WordPress», «это шрифты», если эти зоны не были заранее внесены в ignore-файл.

Если header есть в Figma, он не может быть исключён из diff.

### 7.5 Проверка шапки и глобальной сетки

Header, navigation, topbar, logo, contacts, icons и search считаются частью макета, если они присутствуют в Figma.

Агент обязан сверять:

- высоту header;
- x/y логотипа;
- размеры логотипа;
- пункты меню;
- расстояния между пунктами меню;
- блок адреса и телефона;
- иконки справа;
- высоту hero от верхнего края страницы;
- отступ между header и hero;
- ширину общего container;
- gap между колонками;
- размер левой и правой колонок.

Нельзя писать «основная масса расхождений — это шапка/меню WordPress», если шапка есть в макете. В этом случае шапка является обязательным блоком проверки.

### 7.6 Typography rendering

Перед сдачей агент обязан проверить не только `font-size`, но и полный набор текстовых параметров:

- `font-family`;
- `font-weight`;
- `font-size`;
- `line-height`;
- `letter-spacing`;
- `text-transform`;
- `font-style`;
- наличие подключённого webfont;
- fallback font;
- сглаживание;
- ширину текстового блока;
- количество строк;
- переносы строк.

Если текст в Figma занимает 3 строки, а на сайте 2 или 4 строки без согласованного адаптива — это `FAIL`.

Если шрифт не подключён и браузер использует fallback, это `FAIL`, а не допустимое отличие.

## 8. Fail-gate перед сообщением «готово»

Агент не имеет права писать «готово», если:

- не заполнен паспорт проекта;
- не создан контрольный пакет `figma-audit/`;
- есть ошибки в `figma_assets_report.json`;
- есть хотя бы один `fail` в таблице проверки;
- есть незагруженный SVG/ассет;
- есть визуальный элемент без соответствующего Figma node id;
- есть элемент из Figma, которого нет на сайте;
- есть элемент на сайте, которого нет в Figma;
- не сделаны скриншоты сравнения;
- не создан `comparison-report.md`;
- не создан `hero-layout-report.md` для первого экрана;
- visual diff больше 7%;
- visual diff больше 3% и нет ручного объяснения в отчёте;
- DOM-замеры `pass`, но visual diff `fail`;
- canvas Figma reference и Site screenshot не совпадает;
- header есть в Figma, но не проверен;
- structural nodes не попали в проверку.

Разрешённые финальные статусы:

```text
СДАНО — все проверки pass.
НЕ СДАНО — есть fail, требуется исправление.
ТРЕБУЕТ УТОЧНЕНИЯ — невозможно продолжить без данных.
```

## 12. Специальное правило для ошибки с декоративным крестом и плюсом

Эта ошибка считается эталонным примером критического нарушения регламента.

Нельзя делать так:

```text
Figma: зелёный декоративный крест/звезда
Site: белый круг с плюсом
```

Почему это ошибка:

- изменён тип элемента;
- изменена форма;
- изменён цвет;
- изменена визуальная роль;
- декоративный элемент стал похож на кнопку;
- нарушен состав блока;
- блок визуально перестал соответствовать макету.

Как должно быть:

```text
Figma: VECTOR / GROUP / INSTANCE с зелёным декоративным крестом
Site: тот же SVG/ассет из Figma, в той же позиции, с тем же размером, цветом и opacity
```

Проверка перед сдачей:

```text
- decorative icon node id найден: pass/fail
- decorative icon экспортирован из Figma: pass/fail
- decorative icon подключён на сайте: pass/fail
- форма совпадает: pass/fail
- цвет совпадает: pass/fail
- позиция совпадает ±4 px: pass/fail
- не заменён на другую иконку: pass/fail
```

Если хотя бы один пункт `fail`, блок не сдаётся.

## 13. Финальное правило

Агент обязан работать по принципу:

```text
Не уверен → остановись.
Нет node id → не верстай на глаз.
Нет ассета → не заменяй похожим.
Нет layout skeleton → не заявляй совпадение.
Разный canvas → не сравнивай.
Visual diff fail → работа не сдана.
DOM pass + visual diff fail → работа не сдана.
Есть fail → не пиши «готово».
```

Универсальный порядок проверки:

```text
Сначала совпадает canvas.
Потом совпадает layout skeleton.
Потом совпадают контейнеры.
Потом совпадают тексты, иконки и декоративные элементы.
Если visual diff высокий — работа не сдана, даже при успешных DOM-замерах.
```

Регламент создан для контроля качества вёрстки сайтов и лендингов по Figma и предотвращения типовых ошибок: подмены элементов, изменения геометрии, потери декоративных ассетов, вёрстки на глаз, неполного DOM-покрытия, неверного масштаба сравнения и сдачи без проверки.
